Demo 简介

是什么 · 面向谁 · 核心功能

是什么

索克家居是一个 AI 驱动的全链路智能家居建造平台，已实现可运行的后端 API + 交互式 Demo 页面。后端基于 FastAPI 构建，部署 7 个专业 AI Agent（Orchestrator / Designer / Budget / Procurement / Construction / Settlement / Voice），覆盖注册→设计→BOM→预算→采购→施工→结算全流程。前端 Demo 页面为一个自包含 HTML 单页应用，支持真实 API 与 Simulated Mode 双模式自动切换。

主要功能（已实现）

  1. AI Agent 全链路自治：8 个专业 Agent 混合路由（关键词匹配 + DeepSeek LLM），覆盖设计→预算→采购→施工→质检→结算→客服全流程，LLM 不可用时自动 fallback mock 模式
  2. 交互式 Demo 页面：户型平面图动态渲染 + AI 对话（6 Agent 切换）+ 预算可视化仪表盘 + E2E 一键全链路验证

  4. 多端覆盖：Web 管理后台（10 Tab SPA）+ 统一设计台（2D CAD + Three.js 3D）+ Flutter 移动端 App

Demo 创作思路

灵感来源 · 痛点 · 方向判断 · 关键取舍

灵感来源

中国住宅装修市场 4 万亿规模[1]，但数字化程度极低。业主从毛坯到入住经历六个完全断裂的环节：测量（手动卷尺+草图）→ 设计（AutoCAD/SketchUp/3ds Max 碎片化）→ 选材（跑建材市场）→ 采购（人工比价）→ 施工（微信群沟通）→ 验收（凭经验无标准），预算常超 20–40%。

三项技术在过去 3 年集中突破：Apple RoomPlan（LiDAR 空间扫描）[2]、OpenCascade.js（WASM 几何引擎移动端可用）[3]、LLM Agent 框架（AI 自主运营替代人工）。这是用 AI 重构装修行业的窗口期。

想解决的问题

痛点

传统装修的六大断裂

  • 测量靠手动卷尺+草图，耗时 60+ 分钟
  • 设计用碎片化工具，反复导入导出
  • 选材跑建材市场，信息高度不对称
  • 采购靠人工比价，效率极低
  • 施工靠微信群沟通，变更全凭口头
  • 验收凭经验无标准，纠纷频发
方案

AI 全链路一体化

  • LiDAR 空间扫描 3 分钟完成量房（20x 提效）
  • AI Agent 10 分钟出方案 vs 传统 2 天
  • 215 个物料 SKU 自动匹配 BOM
  • 12 家供应商 AI 自动询比价
  • 8 阶段标准化施工 + AI 质检
  • 全流程数据透明，预算偏差 ≤ 5%

为什么做这个方向

  • 市场空白明确：23 维度竞品对比[5][6][7][8][10][11]显示，无一竞品覆盖超过 40%，AI Agent 自治运营是所有人缺失的能力
  • 技术可行性已达标：WASM 几何引擎[3]、多模态 LLM[9]、WebSocket 实时推送 — 三项关键依赖已成熟
  • 社会价值巨大：每 1% 行业效率提升，对应约 400 亿社会成本节约；AI 质检+透明比价可减少 70% 装修纠纷

开发心得

挑战 1：AI Agent 混合路由设计

纯 LLM 调用成本高且延迟大，纯规则匹配灵活性不足。最终采用混合架构：Orchestrator Agent 先用关键词规则快速分类意图（design / budget / procurement / construction / general），命中后调用对应专业 Agent；LLM 调用失败时自动 fallback 到 mock 模板，保证服务可用性。关键坑点：DeepSeek API key 为空时必须跳过 LLM 调用直接走 mock，否则请求 401。

挑战 2：SQLAlchemy async 链式预加载

selectinload(BOMItem.material).selectinload(Material.category) 链式加载语法容易写错（误写为 BOMItem.material.category），导致 AttributeError。TRAE 帮我们快速定位并修复了 budget_service 中的此问题。

挑战 3：WebSocket 业务集成

21 个变更端点的 project_id 获取策略不统一：create 端点从请求体取，有 project_id 参数的直接用，子资源（budget_line / log / inspection）需通过父资源查询。TRAE 帮助统一了广播模式，并通过 monkeypatch mock 验证了所有端点的广播触发。

挑战 4：Demo 页面双模式适配

Demo 页面需要同时支持真实后端和离线 Simulated Mode。设计方案：页面加载时自动尝试 /api 和根路径两种方式连接后端，连接成功则使用真实 API，失败则自动回退 Simulated Mode，确保任何情况下 Demo 都可完整体验。
